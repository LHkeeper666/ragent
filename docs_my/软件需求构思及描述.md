# "企业智能知识管理与问答平台"软件系统的需求构思及描述

## 背景介绍

随着企业数字化转型的深入，组织内部沉淀了大量非结构化文档（技术文档、产品手册、规章制度、项目报告等），这些知识资产分散在文件服务器、邮件附件、内部 Wiki 等各处，形成信息孤岛。员工查找所需知识时往往依赖关键词搜索或逐层翻阅，效率低下且难以获取精准答案。

与此同时，大语言模型（LLM）技术近两年快速发展，展现出强大的语义理解和文本生成能力。然而，通用 LLM 存在知识截止时间滞后、无法访问企业内部私有数据、"幻觉"输出不可信等问题。检索增强生成（RAG，Retrieval-Augmented Generation）技术应运而生——它结合了信息检索与 LLM 生成，先从知识库中检索相关文档片段，再交由模型基于检索结果生成可信回答，成为企业落地 AI 应用的主流方案。

当前，越来越多的企业希望构建自己的智能知识管理平台，使员工能够以自然语言提问的方式，快速获取基于内部文档的准确回答。这既是提升组织效率的刚性需求，也是 AI 技术向产业落地的必然趋势。

## 欲解决问题

本软件系统旨在解决企业知识管理中以下核心问题：

1. **知识分散，检索困难**：企业文档存储于不同系统（文件服务器、云盘、邮件、Wiki），缺乏统一的检索入口，员工查找信息需要在多个系统之间切换，耗时耗力。

2. **关键词搜索的局限**：传统全文检索依赖关键词匹配，无法理解用户意图和语义。例如，用户搜索"加班怎么申请"，文档中写的是"考勤异常处理流程"，关键词匹配无法建立关联。

3. **知识复用的低效**：新员工入职需要大量时间查阅文档熟悉业务；客服人员需要记忆数百条 FAQ；技术人员解决问题后经验无法有效沉淀和复用。

4. **重复性答疑负担**：行政、HR、IT 等支持部门经常被问及重复性问题（假期政策、报销流程、设备申请等），人工回答效率低且一致性差。

5. **大模型"幻觉"风险**：直接使用通用 LLM 回答企业内部问题，模型可能编造不存在的政策或流程，不可用于严肃的企业场景。

综上，企业需要一个能够整合内部知识资产、理解用户语义意图、基于真实文档生成可信回答的智能平台。

## 软件创意

本项目的核心创意是：**基于开源 RAG 平台（Ragent AI），通过复用、改造和新增相结合的方式，构建企业智能知识管理与问答系统**。

### 开源项目选型

Ragent（GitHub: nageoffer/ragent）是一个成熟的企业级 Agentic RAG 平台，采用 Java 17 + Spring Boot 3.5 + React 18 技术栈，拥有约 4.4 万行 Java 代码和 1.8 万行前端代码，具备多路检索、意图识别、模型路由、MCP 工具集成等核心能力。该项目的技术栈与团队 Java 后端背景高度匹配，代码规模满足课程要求（>10000 行），架构清晰、扩展点设计完善，适合作为改造基础。

### 复用、改造与新增分析

#### 一、可直接复用的模块

以下模块功能完善、架构合理，直接复用无需改动：

| 复用模块 | 说明 |
|:---|:---|
| framework 基础设施层 | 统一响应体、三级异常体系、双维度幂等、Snowflake ID、上下文透传、AOP 链路追踪、Redis 缓存抽象——均为通用能力，零修改复用 |
| infra-ai 模型抽象层 | ChatClient 接口、流式 SSE 解析、首包探测、三态熔断器、模型选择器——模型供应商无关的抽象，直接复用 |
| 多路检索引擎 | VectorGlobalSearchChannel + IntentDirectedSearchChannel 并行检索 + Deduplication + ReRank 后处理流水线，检索引擎设计成熟，直接复用 |
| 意图识别框架 | IntentTreeFactory + IntentResolver + IntentNodeRegistry 树形意图体系，框架层面无需改动 |
| 会话记忆管理 | ConversationMemoryService + 滑动窗口 + 摘要压缩，机制完善直接复用 |
| 分布式排队限流 | ChatQueueLimiter（Redis ZSET + Lua + Pub/Sub），直接复用 |
| 前端页面框架 | React 18 + Vite 工程框架、路由、状态管理，直接复用 |

#### 二、需要改造适配的部分

以下模块核心逻辑可用，根据企业场景进行定制，也是本期的改造重点：

| 改造项 | 现状 | 改造方向 | 预估代码量 |
|:---|:---|:---|:---|
| 意图树节点配置 | 内置通用的知识/非知识/闲聊分类 | 扩展企业专属意图节点（"HR政策咨询""IT技术支持""财务流程问询"等）；新增"增强回答"混合模式——当知识库检索不足时，LLM 可结合自身通用知识补充回答，并标注信息来源；调整各节点的提示词、topK、温度、检索通道等参数 | ~200 行 |
| Prompt 模板 | 通用 RAG 问答模板 | 定制企业 System Prompt（企业语气风格、回答格式、合规约束）；新增增强回答模板（`answer-chat-kb-augmented.st`），定义知识库优先 + 通用知识补充的融合策略，要求来源可辨识；定制 MCP 工具参数提取提示词，提升参数提取准确率 | ~200 行 |

#### 三、需要新增的功能

以下为开源项目不具备、本期从零开发的功能：

| 新增功能 | 说明 | 预估代码量 |
|:---|:---|:---|
| 数据统计与报表 | 无 | 构建 RAG 质量仪表板：检索命中率统计（按知识库/意图分类）、问答覆盖率分析（识别知识盲区）、高频问题排行、用户活跃度趋势。前端管理后台新增统计页面，后端新增统计查询接口 | ~350 行 |

> 改造和新增合计约 750 行代码。所有改动遵循 Ragent 原有编码风格，通过 Ragent 预留的扩展点（IntentNode、PromptTemplate、REST API 等）注入系统，不侵入原有核心逻辑。

### 改造策略总结

```
原有开源项目              →  复用核心架构 + 基础能力
需要改造适配              →  意图树扩展 + Prompt 模板定制（含增强回答模式）
需要新增开发              →  RAG 质量仪表板与数据统计
最终交付                 →  企业智能知识管理与问答平台
```

这种"复用基座 + 适配改造 + 增量开发"的策略，既充分利用了开源项目的成熟设计，又通过针对性的改造和新增使系统贴合企业实际需求，同时满足课程对代码阅读（泛读 10000+ 行、每人精读 1000+ 行）和维护新增（1000+ 行）的量化要求。

## 系统的组成和部署

### 系统组成

本系统由以下要素构成：

| 组件 | 技术选型 | 说明 |
|:---|:---|:---|
| 前端应用 | React 18 + TypeScript + Vite | 用户问答界面 + 管理后台（22 个页面/组件） |
| 后端服务 | Spring Boot 3.5 + Java 17 | 核心业务逻辑、RAG 编排、REST API |
| 关系数据库 | PostgreSQL + pgvector 插件 | 业务数据存储 + 向量检索能力 |
| 向量数据库 | Milvus（可选，pgvector 亦可） | 大规模向量相似度检索 |
| 缓存 | Redis | 会话缓存、排队限流、分布式信号量 |
| 消息队列 | RocketMQ | 异步索引任务、系统解耦 |
| LLM 服务 | 兼容 OpenAI 协议的模型 API | Embedding、Chat、ReRank 模型调用 |
| MCP 服务 | 独立 MCP Server 进程 | 工具注册、参数提取、工具调用 |

### 架构分层

后端按职责划分为四个 Maven 模块：

- **bootstrap**：业务逻辑、REST API、RAG 编排核心链路
- **framework**：共享基础设施（统一响应体、异常体系、幂等、分布式 ID、上下文透传、链路追踪、缓存抽象）
- **infra-ai**：模型供应商抽象层（ChatClient 接口、流式调用、首包探测、三态熔断器、模型选择器）
- **mcp-server**：独立的 MCP 工具服务进程

### 部署方案

采用 Docker Compose 一键部署，各组件部署关系如下：

- 前端 Nginx 静态托管，反向代理 `/api` 到后端服务（端口 9090）
- 后端 Spring Boot 应用支持水平扩展，多实例通过 Redis Pub/Sub 协同
- PostgreSQL + Redis + RocketMQ 挂载数据卷持久化
- Milvus 作为可选向量检索引擎

## 软件系统的功能描述

### 1. 知识库管理

- 创建、编辑、删除知识库，支持多知识库隔离
- 设置知识库的检索策略（topK、相似度阈值、通道选择）
- 文档上传（PDF、Word、Markdown、TXT 等格式）
- 文档解析、智能分块、向量化入库
- 入库任务监控（Pipeline 节点执行日志、进度追踪）

### 2. 智能问答

- **自然语言提问**：用户以自然语言输入问题，系统实时流式输出回答（SSE）
- **问题重写**：结合上下文自动补全和优化用户提问
- **意图识别**：树形多级意图分类，区分知识类/非知识类/闲聊
- **歧义引导**：意图置信度不足时，主动生成选项引导用户澄清
- **多路检索**：向量全局搜索 + 意图导向搜索并行执行，经去重和 ReRank 排序
- **深度思考模式**：可选开启，生成更详尽、更有推理深度的回答

### 3. 会话管理

- 创建/管理多个会话，支持历史会话检索
- 多轮对话上下文关联
- 会话记忆自动摘要压缩，控制 Token 消耗
- 记忆持久化存储，支持 TTL 过期

### 4. 模型引擎

- 多模型供应商配置（兼容 OpenAI 协议）
- 候选模型优先级调度
- 模型健康检查 + 三态熔断器（CLOSED → OPEN → HALF_OPEN）
- 首包探测：模型切换时缓冲事件，避免输出脏数据
- 自动降级：当前模型不可用时无缝切换

### 5. MCP 工具集成

- 按 MCP 协议注册和发现工具
- LLM 自动提取工具调用参数
- 工具调用结果自动融入回答上下文
- 支持扩展自定义业务工具

### 6. 管理后台

- **仪表板**：系统运行概览、调用统计
- **知识库管理**：文档上传、索引管理、数据集维护
- **意图树编辑**：可视化管理意图分类体系
- **链路追踪**：每次问答的完整执行链路（各环节耗时、输入输出、异常信息）
- **模型管理**：模型供应商配置、模型列表维护
- **用户管理**：用户权限管理、操作日志
- **系统设置**：全局参数配置（检索策略、记忆策略、限流策略等）

### 7. 安全与可靠性

- 基于 Sa-Token 的用户认证与权限控制
- 分布式排队限流：Redis ZSET + Lua 脚本 + Pub/Sub，控制并发防过载
- 全链路追踪（AOP）：每个环节可观测、可回溯
- 三级异常体系（Client/Remote/Service）+ 统一异常拦截

## 可行性及潜在风险

### 技术可行性

- 技术栈均为成熟的主流开源技术（Spring Boot、React、PostgreSQL、Redis），团队具备 Java 后端开发背景，学习曲线平缓
- 基础项目 Ragent AI 已有完整可运行的代码（4.4 万行 Java + 1.8 万行前端），架构清晰、设计模式运用得当
- 项目配套完整文档和在线演示，可快速搭建本地开发环境

### 条件可行性

- 开发环境要求：16GB 内存即可运行核心组件（后端 + PostgreSQL + Redis）
- LLM API：可接入国内模型供应商（如 DeepSeek、通义千问等兼容 OpenAI 协议的 API）或本地部署开源模型
- 团队 4 人，按改造任务分工：

| 角色 | 人数 | 负责内容 |
|:---|:---|:---|
| 组长/后端-核心链路 | 1 人 | 整体架构把控、代码评审、Git 分支管理、意图树扩展、Prompt 模板定制 |
| 后端-新增功能 | 1 人 | 用户反馈闭环、数据统计报表、批量导入导出、知识图谱可视化 |
| 后端-MCP与权限 | 1 人 | MCP 工具对接企业 API、知识库权限改造、文档入库 Pipeline 扩展 |
| 前端与文档 | 1 人 | 前端多语言支持、新增页面开发、功能测试、技术文档撰写、答辩 PPT 准备 |

### 时间可行性

- 课程总课时 48 课时（任务一 8 课时 + 任务二 14 课时 + 任务三 8 课时 + 任务四 18 课时）
- 基于成熟开源项目进行学习和改造，而非从零开发，时间可控
- 各阶段任务明确，交付物清晰，可在规定时间内完成

### 潜在风险与应对

| 风险 | 影响 | 应对措施 |
|:---|:---|:---|
| LLM API 不稳定或限流 | 影响问答功能可用性 | 利用多模型路由 + 自动降级机制，配置备用模型 |
| 向量检索精度不足 | 回答质量下降 | 调整分块策略、topK 参数、ReRank 模型，持续优化 |
| 高并发下性能瓶颈 | 响应延迟增加 | 利用排队限流控制并发，水平扩展后端实例 |
| 开源项目代码理解难度 | 影响代码阅读进度 | 借助项目文档、社区资源（CSDN、Stack Overflow）、AI 辅助工具加速理解 |
| 团队协作冲突 | 代码合并冲突、进度不齐 | 使用 Git 分支管理，定期同步进度，明确分工边界 |
